# 06 Notification 隔离与降级

## 修复目标

- 避免 `notification.ts` 继续使用全局单文件，产生跨 agent / 跨 DAG 串数据风险。
- 将其降级为与当前 DAG 上下文一致的隔离存储，而不是潜在主链路状态源。

## 修复思路

- 通知文件路径改为按当前 `agent_id + dag_id` 计算。
- 不在模块加载时固化 `WORKSPACE_DIR`，而是在每次读写时动态解析。
- 保留接口不变，避免影响现有测试和辅助脚本。

## 测试方案

- 新增回归：同一个 task_id 在两个不同 DAG 下写通知，验证互不干扰。
- 保留现有通知测试，验证基本行为不回归。

## 预期结果

- 通知模块至少具备与 DAG 数据一致的基本隔离能力。
- 即使未来重新接回主链路，也不会天然跨 DAG 串数据。
