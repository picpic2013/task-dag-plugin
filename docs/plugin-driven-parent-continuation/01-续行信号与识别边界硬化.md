# 01 续行信号与识别边界硬化

## 开发目标

把 `subagent_ended` 从“尽量恢复父会话”的软逻辑，改成：

1. 只要命中 task-dag 管理对象，就必须产出 continuation 信号。
2. continuation 的成立不依赖 completion 发给谁，也不依赖 `sessions_send` 是否成功。
3. 普通 subagent 不被误识别成 task-dag managed run。

## 开发思路和步骤

### 总体思路

先把“什么情况下应该触发父 agent 继续”定义硬，再去看如何唤醒父会话。

关键原则：

1. “task-dag managed run ended” 是 continuation 的一等事实。
2. `sessions_send` 只是唤醒优化，不是 continuation 成立条件。
3. 普通 subagent 只要没有命中 task-dag 管理对象，就必须完全忽略。

### 开发步骤

1. 收紧 `subagent_ended` 的识别顺序：
   - 优先 `run_id -> session run`
   - 再 `session_key + active binding`
   - 再 `session_key + 单一 active assignment`
   - 最后才允许 `requester_session_key + requester scope` 做辅助恢复
2. 明确“命中 task-dag 管理对象”的判定条件：
   - 命中正式 binding
   - 或命中单一有效 assignment
   - 或命中已登记 session run 且可恢复 parent/dag 上下文
3. 让 `subagent_ended` 在命中后无条件写出权威 continuation 记录。
4. 把“外部消息是否已经发出”从 continuation 语义里剥离。
5. 保持对普通 `sessions_spawn` / 普通 `subagent_ended` 的完全兼容：
   - 不写 runtime 元数据
   - 不写 continuation
   - 不唤醒父会话

## 测试方案

1. task-dag 单任务 run 完成：
   - 触发 `subagent_ended`
   - 检查 continuation 记录必然存在
2. worker session assignment run 完成：
   - 检查 assignment 被消费
   - 检查 continuation 记录存在
3. 普通 subagent ended：
   - 检查不写 continuation
   - 检查不推进 DAG
4. 同一 requester session 下普通 run 与 task-dag run 共存：
   - 检查普通 run 不会误触发 task-dag continuation
5. `sessions_send` 失败场景：
   - 检查 continuation 记录仍然存在

## 测试预期结果

1. 只要是 task-dag 管理的 ended，就一定产生 continuation 事实。
2. continuation 不再依赖 completion 外部投递成功。
3. 普通 subagent 不会被误接管。
4. 父会话恢复的前提从“消息发没发出去”切换为“子 run 是否被 task-dag 收口”。
