# 02 父会话恢复注入与消费模型重构

## 开发目标

让父会话在被触发后的下一轮，稳定地优先继续 DAG，而不是先自由推理或先回复用户。

本里程碑要解决：

1. 父会话虽然被唤醒，但 continuation 指令仍然不够强。
2. `before_prompt_build` 当前更像文本提醒，而不是 continuation 上下文注入。
3. `task_dag_continue` 还没有完全围绕“消费 continuation scope”来表达。

## 开发思路和步骤

### 总体思路

把父会话恢复分成两层：

1. 唤醒层：让父 session 有机会进入新一轮
2. 注入层：让父 session 在新一轮里优先继续 DAG

唤醒不保证一定成功，但只要进入新一轮，注入必须把 continuation 语义前推到最前。

### 开发步骤

1. 强化 `before_prompt_build` 注入内容：
   - 明确 `dag_id`
   - 明确 `run_id`
   - 明确 `task_ids`
   - 明确 `newly_ready_task_ids`
   - 明确优先级：先 `task_dag_continue`，再推进任务，最后才用户回复
2. 让注入内容围绕“待消费 continuation scope”组织，而不是简单拼接提示文本。
3. 收紧多条 continuation 并存时的注入方式：
   - 同一父会话下可列出多个待继续 scope
   - 但每个 scope 仍可单独消费
4. 让 `task_dag_continue` 更清晰地表达：
   - 当前消费的是哪个 scope
   - 继续等待、推进下游、用户回复分别对应什么条件
5. 保持父 agent 的运行模型明确：
   - 不是阻塞等待子 agent
   - 而是当前轮返回，后续由 continuation 新一轮继续

## 测试方案

1. 单个 continuation scope：
   - 检查 `before_prompt_build` 注入明确 scope
2. 多个 continuation scope 并存：
   - 检查注入不混乱
   - 检查每个 scope 仍可单独消费
3. 父会话恢复轮次：
   - 模拟新一轮 prompt build
   - 验证注入优先要求 `task_dag_continue`
4. `sessions_send` 未成功但父 session 后续有新一轮：
   - 检查 `before_prompt_build` 仍可恢复 continuation

## 测试预期结果

1. 父会话恢复后的第一优先动作是继续 DAG。
2. continuation 注入包含明确 scope，而不是模糊提醒。
3. 父会话可以在不依赖阻塞等待的情况下稳定继续执行任务。
4. 唤醒失败不会让 continuation 彻底丢失。
