# 04 测试矩阵与文档收口

## 开发目标

把这一轮 feature 的行为边界通过测试和文档固定下来，避免后续又退回到“completion 发给谁”与“父会话能否继续”混在一起的状态。

## 开发思路和步骤

### 总体思路

先用测试把设计收死，再用 README / SKILL / MIGRATION 把外部使用方式写清楚。

文档要表达的核心原则只有一条：

- completion 的外部投递路径不是父 agent 继续执行 DAG 的前提；
- task-dag 的前提是：managed subagent ended 后，插件一定产出并保留 continuation。

### 开发步骤

1. 补完整体测试矩阵：
   - task-dag 单任务 run
   - shared worker assignment run
   - 并发 ended
   - 唤醒失败
   - 普通 subagent 兼容
2. 补针对 `before_prompt_build` 的恢复注入测试。
3. 补针对 `task_dag_continue` 的 scope 消费测试。
4. 更新文档：
   - README
   - SKILL
   - MIGRATION
   - 新 feature README
5. 收口术语，避免文档继续混用：
   - “阻塞等待”
   - “恢复旧工具栈”
   - “父子 agent 直接通话”

## 测试方案

1. 回归测试：
   - 现有 task-dag 核心测试全部通过
2. 新增行为测试：
   - managed ended 必有 continuation
   - 普通 ended 无 continuation
   - `sessions_send` 失败仍可恢复
   - 多 continuation scope 并存仍可消费
3. 文档一致性检查：
   - README / SKILL / MIGRATION 都采用同一运行模型表述

## 测试预期结果

1. 这一轮 feature 的运行模型被测试固定。
2. 用户和 agent 都能从文档中明确理解：
   - 父 agent 是“返回后等触发”，不是“阻塞等待”
   - 普通 subagent 不会进入 task-dag continuation 链路
3. 后续继续迭代 completion 路由时，不会破坏本轮已固定的 parent continuation 语义。
