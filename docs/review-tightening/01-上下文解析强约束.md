# 01 上下文解析强约束

## 问题所在

当前工具层和 binding 层仍允许在缺少显式 `agent_id/dag_id` 时回退到全局 `currentAgentId/currentDagId`。这会把“上一次调用留下的上下文”带到下一次调用，导致多 agent、多 DAG、hook 并发时串目录。

## 修复思路

- 收紧 `setExecutionContext()`：
  - 只允许以下几类确定性来源：
    - 显式 `parent_agent_id`
    - 显式 `agent_id`
    - 显式 `context.agent.id / context.agentId / context.session.agentId / context.runtime.agentId`
    - 显式 `dag_id` 且上下文中可确定当前 agent
  - 否则直接报错，不再静默继承旧全局上下文。
- 收紧 binding 层 `resolveContext()`：
  - 缺少 `agentId` 或 `dagId` 时抛错，而不是回退到全局默认值。
- 为继续保留“当前 DAG 内操作”的体验，允许“已创建当前 DAG 且当前 agent 明确”的调用路径，但必须基于本次调用已设置好的上下文，不允许跨调用猜测。

## 测试

- 新增测试：无显式上下文时调用 binding API，应直接抛错。
- 新增测试：无显式上下文时调用 `claimTaskExecution()`，应返回上下文缺失错误。
- 新增测试：同一进程中切换过 agent/dag 后，后续缺少显式上下文的操作不能偷偷继承旧值。

## 测试预期结果

- 所有缺少确定性上下文的关键入口都会失败并给出明确错误。
- 需要上下文的调用只能命中本次明确指定的 agent/dag。
