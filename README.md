# Task DAG (plugin)

Task DAG is an OpenClaw plugin for orchestrating multi-step work as a DAG.
It gives agents explicit task state, subagent bindings, hook-driven cleanup, and deterministic parent-session continuation.

## What This Plugin Is For

Use this plugin when an agent needs to:

- split work into explicit tasks with dependencies
- send some tasks to subagents and keep others in the parent session
- continue the parent session after subagent completion without relying on model memory
- keep task state on disk instead of reconstructing it from chat history

This plugin is designed around OpenClaw's runtime model:

- agents call native `sessions_spawn` and `sessions_send`
- the plugin tracks DAG state, spawn intents, bindings, pending events, and continuation
- the parent agent does not block on a child run; it returns, then resumes in a later round

## Status

Current runtime model:

- managed spawn identity uses a short protocol label: `task-dag:<10-char-base36>`
- `subagent_spawned` binds `run_id` back to the prepared task
- `subagent_ended` closes the task by `run_id`
- continuation is recorded as `resume_requested`; `sessions_send` is only a wake-up optimization

## Enable

Enable the plugin in `openclaw.json`:

```json
{
  "plugins": {
    "entries": {
      "task-dag": {
        "enabled": true
      }
    }
  }
}
```

Allow the tools on the agents that should use them:

```json
{
  "agents": {
    "list": [
      {
        "id": "main",
        "tools": {
          "allow": ["task-dag"]
        }
      }
    ]
  }
}
```

## Config

```json
{
  "plugins": {
    "entries": {
      "task-dag": {
        "enabled": true,
        "config": {
          "heartbeatInterval": "30s",
          "autoProceed": false,
          "maxBindingAttempts": 3
        }
      }
    }
  }
}
```

Config keys:

- `heartbeatInterval`: plugin heartbeat interval
- `autoProceed`: reserved for continuation policy
- `maxBindingAttempts`: safety cap for rebinding one task; the default is `3`

## Execution Modes

A `ready` task should use exactly one of these paths.

Parent executes directly:

```text
task_dag_claim -> task_dag_progress -> task_dag_complete
```

Subagent executes:

```text
task_dag_spawn -> sessions_spawn(using spawn_plan unchanged) -> task_dag_continue
```

Do not mix them for the same task.
Do not `claim` a task and then `spawn` it.

## Quick Start

### 1. Create a DAG

```bash
task_dag_create agent_id="main" name="Example" tasks=[
  {id:"t1",name:"Research"},
  {id:"t2",name:"Write",dependencies:["t1"]}
]
```

### 2. Parent executes a task

```bash
task_dag_claim agent_id="main" task_id="t1" executor_type="parent"
task_dag_progress agent_id="main" task_id="t1" progress=50 message="halfway"
task_dag_complete agent_id="main" task_id="t1" output_summary="done"
task_dag_continue agent_id="main" task_id="t1"
```

### 3. Send a task to a subagent

```bash
task_dag_spawn agent_id="main" requester_session_key="agent:main:feishu:group:xxx" task_id="t2" task="Finish the analysis" target_agent_id="worker"
```

Then call native `sessions_spawn` with the returned `spawn_plan` unchanged.

Important:

- OpenClaw `sessions_spawn` expects `agentId`
- do not rewrite it as `target_agent_id`
- do not override the generated `label`

After the child run finishes, continue from the parent session with:

```bash
task_dag_continue agent_id="main" task_id="t2"
```

### 4. Reuse a worker session

```text
first run: task_dag_spawn -> sessions_spawn
next run: task_dag_assign -> sessions_send -> task_dag_continue
```

Worker reuse is supported, but new-task recognition in later rounds still depends on `task_dag_assign`.
It is not inferred from free-form chat alone.

## Runtime Rules

The plugin is intentionally strict.

- non-`main` agents must pass `agent_id` explicitly
- `task_dag_spawn` should include `requester_session_key` if parent continuation matters
- `task_dag_continue` requires an explicit scope: `run_id`, `session_key`, `task_id`, or `task_ids`
- unmanaged subagents are ignored
- only protocol labels generated by the plugin are treated as managed spawn candidates
- if a task exceeds `maxBindingAttempts`, rebinding fails fast instead of silently accumulating bad state

## Data on Disk

Per-DAG runtime state is stored under the agent workspace, for example:

```text
.openclaw/workspace-<agent>/tasks/<dag_id>/
├── dag.json
├── spawn-intents.json
├── assignment-intents.json
├── session-runs.json
├── task-bindings.json
└── pending-events.jsonl
```

## Design Notes

The most important design choice is that continuation does not depend on who receives the completion message first.

What matters is:

1. a managed child run is recognized in `subagent_spawned`
2. its `run_id` is bound to a task
3. `subagent_ended` closes that task and emits continuation state
4. the parent session consumes that continuation in a later round

This is why the parent model is:

- logically waiting
- operationally returning
- later resuming through continuation
